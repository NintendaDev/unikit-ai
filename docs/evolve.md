# Memory & Skill Evolution

Every `/unikit-fix` leaves behind a **patch** - a structured record of what went wrong and how it was fixed. `/unikit-evolve` processes patches incrementally, analyzes your project's tech stack and conventions, and turns recurring patterns into **rules** that make skills smarter. The more you fix, the better the evolution.

## The Learning Loop

```
  /unikit-fix         patches/             /unikit-evolve          skill rules
 ┌──────────┐      ┌──────────────┐      ┌──────────────┐      ┌──────────────┐
 │ Find bug │ ───▶ │ Save patch   │ ───▶ │ Analyze new  │ ───▶ │ Classify:    │
 │ Fix it   │      │ with root    │      │ patches +    │      │ code rules → │
 │ Log it   │      │ cause + tags │      │ project ctx  │      │   RULES.md   │
 └──────────┘      └──────────────┘      └──────────────┘      │ skill issues │
      ▲                                                        │   → context  │
      │                                                        └──────┬───────┘
      │                                                               │
      └──────────── better /unikit-fix, /unikit-implement ◀──────────┘
```

Each iteration of this loop makes the AI more aware of your project's pitfalls. After 5-10 fixes, `/unikit-evolve` has enough signal to add targeted guards - not generic advice, but rules derived from real bugs in your codebase.

## Two Classification Categories

Unlike generic tools that treat all improvements the same, `/unikit-evolve` classifies patches into two distinct categories:

### 1. Code and Architecture Rules

Problems in code, architectural patterns, recurring mistakes:
- Written to `RULES.md` (project-specific rules with highest priority)
- Can later be migrated to permanent `memory/stack/` via `/unikit-memory`
- Example: "Always throw when DiResolver.Resolve returns null in NodeCanvas OnInit"

### 2. Skill Workflow Issues

When a skill itself worked incorrectly or gave bad recommendations:
- Written to `.unikit/skill-context/<skill>/SKILL.md` as overrides
- Override specific skill behavior without modifying the base SKILL.md
- Example: "When creating NodeCanvas tasks, always check for Zenject DiResolver availability"

## Incremental Processing

State file:

```
.unikit/evolutions/patch-cursor.json
```

- **No cursor:** first evolve run reads all patches
- **Cursor present:** evolve reads new patches newer than cursor, plus a small overlap window (reruns help catch missed points)
- **Full rescan:** delete cursor file and rerun `/unikit-evolve`

## Skill-Context: Project-Specific Skill Instructions

### What is Skill-Context

Built-in `unikit-*` skills ship with UniKit AI and get overwritten on update. To keep project-specific rules stable, `/unikit-evolve` writes them to a separate location:

```
.unikit/skill-context/<skill-name>/SKILL.md
```

Key properties:
- **Survives `unikit-ai update`** - lives in your project, not in the package
- **Higher priority than base rules** - when a skill-context rule conflicts with base SKILL.md, the skill-context rule wins
- **Cumulative** - each `/unikit-evolve` run adds new rules, updates existing ones, and removes stale ones

### In Practice: Use /unikit-skills-context Instead

In ~99% of cases `/unikit-evolve` will not produce skill-context improvements - patch information almost always relates to code rules, tech stack, and architecture, not to skill workflow issues. If you need to explicitly override how a specific skill behaves in your project, use `/unikit-skills-context` directly:

```
# Add a rule to a specific skill
/unikit-skills-context review "Always check null-check symmetry first"

# Interactive mode - will ask which skill and what rule
/unikit-skills-context

# Validate existing skill-context rules against updated base skills
/unikit-skills-context validate
```

See [Skills Reference](skills.md) for full `/unikit-skills-context` documentation.

### Context File Format

Skill-context files are always written in English, regardless of the project's language:

```markdown
# Project Rules for /unikit-fix

> Auto-generated by `/unikit-evolve`. Do not edit manually.
> Last updated: 2026-03-15 14:30
> Based on: 8 analyzed patches

## Rules

### Null-check for DiResolver.Resolve in NodeCanvas
**Source**: 2026-03-10-09.15.md, 2026-03-12-16.45.md
**Rule**: Always throw InvalidOperationException when DiResolver.Resolve
returns null for mandatory dependencies in NodeCanvas OnInit/OnExecute.

### UniTask CancellationToken propagation
**Source**: 2026-03-13-10.00.md
**Rule**: Every async method in ActionTask/ConditionTask must accept and
propagate CancellationToken from the graph owner's destroyCancellationToken.
```

### How Skills Use Context

Every `unikit-*` skill reads its own context file at startup:

```
Read .unikit/skill-context/<skill-name>/SKILL.md
```

If the file exists, the skill treats its rules as project-level overrides - more specific context always wins over general instructions.

## Running /unikit-evolve

### Usage

```
/unikit-evolve
```

Reads all new patches incrementally and proposes rules across all skills.

### Workflow Overview

| Step | What happens |
|------|--------------|
| **1: Read patches** | Glob `.unikit/patches/*.md`, apply incremental cursor + overlap window (tail-5 anti-miss guard). Extract prevention points, root causes, and tags from each patch. A single patch can produce multiple independent prevention points |
| **2: Load context** | Read `DESCRIPTION.md` (tech stack), `RULES.md` (current rules), `RULES_INDEX.md` (knowledge base index) |
| **3: Build registry** | Build flat list of all prevention points. Classify each as `code` (patterns, null-checks, async, DI - destination: RULES.md) or `workflow:<skill>` (delegation, compilation checks, parallelism - destination: skill-context) |
| **4: Filter covered** | Cross-check each point against existing rules. Code rules checked against RULES.md + knowledge base via RULES_INDEX.md. Workflow rules checked against skill-context + base SKILL.md. Only truly uncovered points survive |
| **5: Present candidates** | Group uncovered points by type, show source patch and proposed rule text. User chooses: apply all, pick individually (batches of 4), or skip all |
| **6: Add accepted rules** | Code rules added via `/unikit-rules` (handles dedup, section placement). Workflow rules written directly to `.unikit/skill-context/<skill>/SKILL.md` |
| **7: Log & advance cursor** | Save evolution log to `.unikit/evolutions/YYYY-MM-DD-HH.mm.md` with stats and added rules. Advance cursor to newest processed patch. Suggest `/clear` or `/compact` to free context |

## How Rule Verification Works

`/unikit-evolve` does not add rules blindly - every candidate passes through a verification chain before it reaches the user.

### Code rules - delegation to /unikit-rules

Accepted code rules are not written to `RULES.md` directly. Instead, `/unikit-evolve` invokes `/unikit-rules` for each one. The skill handles the full verification pipeline:

1. Reads `RULES_INDEX.md` and identifies knowledge base files that overlap with the rule's topic
2. Reads the relevant knowledge base files (not all - only those matching the topic)
3. Cross-checks against existing `RULES.md` entries for semantic duplicates
4. Determines the relationship - already covered, contradicts, extends, or new topic
5. Places the rule in the correct `RULES.md` section (or creates a new one)

If a rule is already covered by the knowledge base or `RULES.md`, `/unikit-rules` reports it and skips - `/unikit-evolve` notes the result and moves on.

### Workflow rules - internal cross-check

Workflow rules (`workflow:<skill>` type) are verified by `/unikit-evolve` itself in Step 4:

1. Checks `.unikit/skill-context/<skill>/SKILL.md` for semantic duplicates
2. Checks the base skill at `<skills_dir>/<skill>/SKILL.md` for existing coverage
3. Only uncovered points reach the user as candidates

### Standalone stale rule validation

To check whether existing skill-context rules have become stale after a `unikit-ai update` (base skills now cover them), use a separate command:

```
/unikit-skills-context validate
```

This compares every rule in `.unikit/skill-context/` against updated base skills and offers keep/remove/rewrite decisions per rule. See [Skill-Context](#skill-context-project-specific-skill-instructions) above for details.

## When to Evolve

- **After 5-10 fixes** - enough patches for meaningful pattern detection
- **After changing tech stack** - new framework means new pitfalls to learn
- **When the same bug type keeps recurring** - signal that skills need a targeted guard
- **After a `unikit-ai update`** - stale rule cleanup removes rules now covered upstream
