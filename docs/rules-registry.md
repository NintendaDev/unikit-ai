# Rules Registry

UniKit AI fetches all knowledge-base rules - both **core** and **stack** - from a remote registry, not from the npm package. This lets rules be versioned and updated independently of UniKit releases.

## Overview

The **official registry** (`NintendaDev/unikit-ai-rules`) ships community-maintained rules for the most popular Unity frameworks.

**Your own registry** is just as easy to set up - a local folder on disk or a private GitHub repo. Once configured, it works identically to the official one and lets you carry your team's rule library from project to project without copy-pasting files.

### Registry and `/unikit-memory`

The registry is the primary source `/unikit-memory` consults when creating new agent memory documents:

1. `/unikit-memory` runs `unikit-ai rules list` to check the registry catalog.
2. If a rule for the requested framework exists - official or in your custom registry - the skill offers to download and install it directly. This is the fastest path: battle-tested rules arrive in one step.
3. If the registry has no match, or if you prefer to start fresh, you can decline the download. The skill then generates the rule from scratch using its own analysis algorithms - interviewing you about the framework, reading existing code, and synthesizing the document.

Both paths produce rules in `.unikit/memory/`. The registry path is faster for well-covered frameworks; the generate path is the right choice for proprietary or niche tech.

## How It Works

### Rule Types

| Type | Description | Install Behavior |
|------|-------------|-----------------|
| **Core** | Foundational rules loaded on every task - code style, design principles, folder structure, performance, testing | Always installed. Required for the framework to function correctly. |
| **Stack** | Framework-specific rules: DOTween, Zenject, R3, NodeCanvas, Odin, etc. | Installed on demand - only rules matching your tech stack. |

#### Core rules

There are exactly **6 core rules** with fixed IDs. A custom registry may override any of them, but cannot add or remove entries from this set:

| Rule ID | Purpose |
|---------|---------|
| `code-style` | Naming conventions, access modifiers, member ordering, class structure |
| `design-principles` | SOLID, GRASP, KISS/DRY, SRP decision framework |
| `folders-structure` | Project folder layout, file placement conventions |
| `performance` | Pooling, caching, ZLinq, hot path optimization, mobile constraints |
| `testing` | NUnit, AAA pattern, test doubles, boundary conditions |
| `pipeline` | Spec-driven workflow contract shared across all pipeline skills |

These rules are fully customizable - you can rewrite any of them in your own registry to reflect your team's pipeline, naming conventions, or architectural constraints. If a rule is missing from the custom registry, it falls back to the official version automatically. This means you only need to provide the rules you actually want to override; the rest are inherited from the official registry.

### Transport Chain

Resolution happens **per engine** through a three-level fallback:

```
1. Primary registry (user-configured URL or local path, if any)
   ├── manifest has your engine? → use primary for ALL rules (core + stack)
   └── engine missing or fetch failed → fall through

2. Official registry (raw.githubusercontent.com/NintendaDev/unikit-ai-rules/main)
   ├── engine exists → use official
   └── fetch failed → fall through

3. Bundled snapshot (shipped inside the npm package as rules-registry/)
   └── last-resort fallback - may be older than the live official registry
```

If all three levels are unreachable, `rules install` exits with code 2 (network error).

Resolution is **per-engine** (coarse). If the primary registry covers your engine, all rules for that engine come from it - no per-rule mixing between registries.

### Registry Structure

A registry is a directory with the following layout:

```
my-registry/
├── manifest.json               # Auto-generated by scripts/build-manifest.js
├── scripts/
│   └── build-manifest.js       # Manifest builder (scaffolded by rules registry init)
└── <engineId>/                 # One folder per supported engine
    ├── core/
    │   ├── code-style.md
    │   ├── design-principles.md
    │   ├── folders-structure.md
    │   ├── performance.md
    │   ├── pipeline.md
    │   └── testing.md
    └── stack/
        ├── dotween.md
        ├── zenject.md
        └── references/         # Reference files linked from rule headers
            └── dotween-api-quickref.md
```

Supported engine IDs: `unity`, `godot`, `godot-net`, `unreal-engine-5`.

**You don't need to cover every engine.** A registry is valid if it contains rules for just one engine - or even a subset of core rules for that engine. When `unikit-ai` fetches rules, it reads the engine ID from `.unikit.json` and looks only for that engine's folder in the registry. If the project is initialized for `unity` and the registry contains only a `unity/` folder, everything works correctly - the CLI never requests rules for engines not present in the registry.

**Creating and maintaining a registry is fully automated.** The `/unikit-rules-registry` skill handles the full lifecycle: it scaffolds the registry structure from scratch, seeds it with the rules currently in `.unikit/memory/`, and keeps it in sync as the project's knowledge base evolves. Use it to bootstrap a registry in minutes and to push rule updates without touching the registry files manually.

## Custom Registry Workflow

A **custom local registry** is a file-system clone of the registry structure that you own. It lets you:
- Override official core rules with team- or org-specific versions
- Ship private stack rules not present in the official registry
- Version and review rule changes the same way you version code

Use the `/unikit-rules-registry` skill, which owns three modes: **create**, **update**, and **sync**.

### create - scaffold and seed a new registry

```
/unikit-rules-registry create
```

What happens:
1. Asks where to create the registry (default: `./rules-registry`)
2. Runs `unikit-ai rules registry init <path>` to scaffold the directory and `manifest.json`
3. Copies all rules from `.unikit/memory/core/` and `.unikit/memory/stack/` into the registry (`version: 1.0.0` injected into each rule copy)
4. Regenerates the manifest via `node <path>/scripts/build-manifest.js`
5. Optionally switches the project to the new registry (`registry set` + `rules sync --replace --prune`)

CLI equivalent:

```bash
unikit-ai rules registry init ./rules-registry
unikit-ai rules registry set ./rules-registry
unikit-ai rules sync --replace --prune
```

### update - push memory changes into the registry

After editing rules in `.unikit/memory/` (via `/unikit-memory` or manually), push those changes into the registry:

```
/unikit-rules-registry update
```

What happens:
1. Diffs `.unikit/memory/` against the registry
2. Copies new rules (`version: 1.0.0`) and auto-bumps versions of changed rules (patch / minor / major based on diff size)
3. Cleans up reference files no longer referenced by any rule
4. Regenerates the registry manifest
5. Runs `unikit-ai rules install <changed-ids> --force` to reconcile `.unikit.json` state

> **`/unikit-memory` integration** - when a local file-based registry is active, `/unikit-memory` will prompt after creating or migrating rules:
> *"New rules were added to `.unikit/memory/`. Push them to the registry? Run `/unikit-rules-registry update`."*
>
> Run `/unikit-rules-registry update` when ready; it handles versioning and state reconciliation automatically.

### sync - pull registry changes into memory

After editing rules in the registry directly, or after a collaborator commits rule changes to a shared registry, sync them back:

```
/unikit-rules-registry sync
```

The skill asks which sync mode to use:

| # | Mode | What it does |
|---|------|-------------|
| 1 | **Safe** | Updates only rules already marked `source: registry` whose version changed. Skips locally-modified rules and does not remove anything. |
| 2 | **Replace** | Same as Safe, but also overwrites rules that are up-to-date in version but locally modified on disk. |
| 3 | **Mirror** | Full parity with the registry: replaces all rules (including local modifications) and removes obsolete stack rules that vanished from the manifest. Equivalent to the legacy `--force`. |

CLI equivalents (for use outside the skill):

```bash
unikit-ai rules sync                        # Safe
unikit-ai rules sync --replace              # Replace
unikit-ai rules sync --replace --prune      # Mirror
```

**Direction summary:**

| Mode | Direction |
|------|-----------|
| `create` | memory → registry (initial seed) |
| `update` | memory → registry (incremental push) |
| `sync` | registry → memory (pull) |

## CLI Commands

### `unikit-ai rules list [--json] [--engine <id>]`

List available rules from the registry catalog.

```bash
unikit-ai rules list                 # Human-readable
unikit-ai rules list --json          # JSON for AI skills
unikit-ai rules list --engine godot  # Override engine filter
```

### `unikit-ai rules show <id> [--references]`

Preview a rule from the registry without installing it.

```bash
unikit-ai rules show dotween
unikit-ai rules show aspid-mvvm --references
```

Rule IDs use canonical lowercase-hyphen form. The CLI matches IDs case-insensitively, so `CODE-STYLE` resolves to the same file as `code-style`.

### `unikit-ai rules install [ids...] [--force]`

Install rules from the registry. The command is variadic with two contracts:

- **No arguments** - installs the whitelisted core rule set (used internally by `/unikit` Step 9.2). Fetches the manifest once, writes each missing core rule to `.unikit/memory/core/`, regenerates `RULES_INDEX.md`.
- **With one or more IDs** - installs every listed ID in a single pass, prints an aggregated report with per-rule `✓ installed` / `↻ already installed` / `✗ failed` lines followed by a summary.

```bash
unikit-ai rules install                            # Core whitelist bootstrap
unikit-ai rules install urp dotween unitask        # Variadic install
unikit-ai rules install code-style --force         # Force re-fetch an already-installed rule
```

`--force` overwrites a rule already in state. Without it, installed rules are absorbed as `↻ already installed` without affecting the exit code - safe to re-run for drift recovery.

### `unikit-ai rules sync [--replace] [--prune]`

Reconcile disk state with `.unikit.json` and regenerate `RULES_INDEX.md`. Run after manually editing rule files, or after switching the registry with `rules registry set`.

Flags are composable:

- `--replace` - overwrite registry-sourced rules at the same version AND rules with local modifications. Without it, same-version and locally-modified rules are skipped with a WARN event.
- `--prune` - remove obsolete stack rules that vanished from the registry manifest. Scoped to stack (core rules are whitelist-governed and never pruned automatically).

```bash
unikit-ai rules sync                        # Version-drift refresh only (safe default)
unikit-ai rules sync --replace              # Also overwrite local modifications
unikit-ai rules sync --replace --prune      # Full mirror (old --force equivalent)
```

Sync runs in three phases (shared with `unikit-ai update`):

1. **Phase 1 - disk/state reconciliation.** Untracked `.md` files found in `.unikit/memory/{core,stack}/` are tagged `source: local`; entries missing from disk are dropped from state. IDs are compared case-insensitively so legacy `CODE-STYLE` and canonical `code-style` never double-register.
2. **Phase 2 - registry sync.** Each rule with `source: "registry"` is compared to the manifest version. Newer registry versions replace the installed file; equal versions and locally-modified files are skipped unless `--replace` is set. If the registry version is LOWER than the installed version, sync emits a `phase2:downgrade` log line and still applies the change under `--replace` - so accidental rollbacks are never silent.
3. **Phase 3 - index regeneration.** `.unikit/memory/RULES_INDEX.md` is rebuilt from registry metadata with a disk fallback for user-added rules. Runs on every sync, regardless of flags.

### `unikit-ai rules status [--json]`

Show installed rules with their sources, origins, and versions.

```bash
unikit-ai rules status
unikit-ai rules status --json   # Machine-readable for AI skills
```

### `unikit-ai rules registry [show | set <url> | reset | init [path]]`

Nested command group. The bare form (`rules registry` with no subcommand) is an alias for `rules registry show`. None of the write subcommands trigger a sync - switching the URL and pulling content are two explicit steps.

```bash
unikit-ai rules registry                                        # → rules registry show
unikit-ai rules registry show                                   # Print the configured URL
unikit-ai rules registry show --json                            # JSON output

unikit-ai rules registry set https://example.com/my-rules       # Write URL only (no sync)
unikit-ai rules registry set ./rules-registry                   # Set a local path registry
unikit-ai rules registry reset                                  # Revert to official registry (no sync)

unikit-ai rules registry init ./rules-registry                  # Scaffold a new local registry
unikit-ai rules registry init ../shared/org-rules               # Custom path
```

After `set` or `reset`, the CLI prints a hint block with suggested `rules sync` variants. Run the appropriate sync when ready to pull content from the newly configured registry.

## `.unikit.json` Registry Fields

### `rulesRegistry`

```json
{
  "rulesRegistry": "https://raw.githubusercontent.com/NintendaDev/unikit-ai-rules/main"
}
```

| Value | Meaning |
|-------|---------|
| HTTPS URL | Remote registry served over HTTP (GitHub raw, any CDN) |
| Local path (`./…`, `../…`, `/…`) | Local file-system registry; auto-detected by the CLI as the `fs` transport |
| `null` | Legacy value - maps to the official registry at runtime. New writes never produce `null`. |

Both `unikit-ai init` (when the custom-registry wizard step is declined or skipped) and `unikit-ai rules registry reset` write the official `NintendaDev/unikit-ai-rules` URL verbatim, so new and reset projects always have a concrete URL.

### `rules.installed`

Tracks every rule installed into `.unikit/memory/`:

```json
{
  "rules": {
    "installed": {
      "version": "1.2.3",
      "core": [
        { "name": "code-style",        "source": "registry", "origin": "official", "version": "1.2.0", "installed_hash": "sha256:..." },
        { "name": "folders-structure", "source": "registry", "origin": "primary",  "version": "2.0.0", "installed_hash": "sha256:..." }
      ],
      "stack": [
        { "name": "dotween",   "source": "registry", "origin": "primary", "version": "1.0.0", "installed_hash": "sha256:..." },
        { "name": "my-custom", "source": "local" }
      ]
    }
  }
}
```

**Fields per rule entry:**

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Rule ID in canonical lowercase-hyphen form |
| `source` | `"registry"` \| `"installer"` \| `"local"` | Where the rule came from |
| `origin` | `"primary"` \| `"official"` \| `"bundled"` | Which registry level served the rule (only when `source: "registry"`) |
| `version` | string | Semver from the registry manifest |
| `installed_hash` | string | SHA-256 of installed content, used to detect local modifications |

**`source` values:**

| Value | Meaning |
|-------|---------|
| `"registry"` | Fetched from the registry chain (primary → official → bundled) |
| `"installer"` | Installed during `unikit-ai init` before the object-format schema existed (legacy) |
| `"local"` | Manually added to `.unikit/memory/` and not tracked in any registry |

**`origin` values (only when `source: "registry"`):**

| Value | Meaning |
|-------|---------|
| `"primary"` | Served from the user-configured custom registry |
| `"official"` | Served from `NintendaDev/unikit-ai-rules` |
| `"bundled"` | Served from the npm-bundled snapshot (`rules-registry/`) |

Legacy `string[]` entries (e.g. `"core": ["code-style"]`) are automatically normalized to `{ name, source: "installer" }` on load. The next `rules sync` or `rules install --force` upgrades them to full object entries.

## Setting the Registry

### During `unikit-ai init`

The wizard interactively asks whether to connect a custom registry (confirm → URL input → validate → retry/skip loop). A confirmed, valid URL is stored in `.unikit.json.rulesRegistry`. If declined or skipped, the official URL is written instead.

### After init

```bash
# Switch to a remote custom registry
unikit-ai rules registry set https://raw.githubusercontent.com/my-org/my-rules/main
unikit-ai rules sync --replace --prune

# Switch to a local registry
unikit-ai rules registry set ./rules-registry
unikit-ai rules sync --replace --prune

# Reset to the official registry
unikit-ai rules registry reset
unikit-ai rules sync --replace --prune
```

> **Re-init warning:** `unikit-ai init` on an already-configured project does **not** pre-fill the previous `rulesRegistry` value - the wizard always starts from scratch. Use `unikit-ai rules registry set <url>` to change the registry without re-running `init`.

## Exit Codes

All `unikit-ai rules *` commands share a unified exit-code set:

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Not found (rule ID, config file, etc.) |
| 2 | Network error / registry unreachable |
| 3 | Invalid arguments (bad ID format, relative path) |
| 4 | Operation not permitted (file-exists guards) |
| 5 | Registry validation failed (bad manifest, schema mismatch, engine missing, empty core whitelist) |
| 6 | Registry already initialized at target path (`rules registry init`) |
| 7 | Target path occupied by non-registry files (`rules registry init`) |

## Testing

The `unikit-ai rules *` surface has dedicated smoke tests under `scripts/test-rules-*.sh`. Each file owns one command and runs against fake rule registries in `scripts/test-fixtures/` so assertions stay deterministic without hitting the real bundled snapshot.

### Layout

```
scripts/
├── test-fixtures.sh                     # Shared helpers (assertions, fake registries, counters)
├── test-fixtures/
│   ├── minimal-valid/                   # 1 core + 1 stack rule with references (unity + godot)
│   ├── multi-version/
│   │   ├── v1/                          # unitask v1.0.0 baseline
│   │   └── v2/                          # unitask v2.0.0 upgraded (bumped hash + sentinel)
│   └── corrupted-manifest/              # Invalid JSON → exit 5 on `rules registry set`
├── test-rules-list.sh                   # `rules list` - exit codes, --json shape, --engine override
├── test-rules-show.sh                   # `rules show` - id normalization, --references expansion
├── test-rules-status.sh                 # `rules status` - populated state, registryKind, --check-updates guard
├── test-rules-install.sh                # `rules install` - no-args bootstrap, variadic, --force, drift recovery
├── test-rules-sync.sh                   # `rules sync` - 4 modes × 4 states + 4 regression guards
├── test-rules-registry.sh               # `rules registry` - show/set/reset, no-auto-sync guards, origin fold
├── test-rules-registry-init.sh          # `rules registry init` - scaffold smoke tests
├── test-rules.sh                        # Runner: Parts 1-9 CLI contract + Parts 11-17 per-command tests
└── test-exit-codes.sh                   # Guard matrix: every documented exit code must be exercised
```

### Runner chain

`npm test` → `scripts/test-skills.sh` (main runner) → Part 13 runs `scripts/test-rules.sh`, which in turn chains Parts 11-17 through the per-command files and the exit-code matrix guard. The chain is:

```
npm test
└── scripts/test-skills.sh
    ├── Parts 1-7   skill / subagent / data / code validation
    ├── Part 8      test-update.sh
    ├── Part 9      test-install.sh
    ├── Part 10     test-extensions.sh
    └── Part 13     test-rules.sh
        ├── Parts 1-9   registry module, CLI contract, manifest, bundled snapshot
        ├── Part 11     test-rules-list.sh
        ├── Part 12     test-rules-show.sh
        ├── Part 13     test-rules-status.sh
        ├── Part 14     test-rules-install.sh
        ├── Part 15     test-rules-sync.sh
        ├── Part 16     test-rules-registry.sh
        └── Part 17     test-exit-codes.sh   ← matrix guard
```

### Running tests in isolation

Every per-command file builds `dist/` and runs standalone:

```bash
bash scripts/test-rules-list.sh
bash scripts/test-rules-install.sh
bash scripts/test-rules-sync.sh
bash scripts/test-rules.sh                # CLI contract + all per-command children
bash scripts/test-exit-codes.sh           # Matrix guard (reads the other files)
```

### Writing new scenarios

1. Pick or create a fixture under `scripts/test-fixtures/`. Keep fixtures minimal - they exist to exercise specific code paths, not to mirror real content.
2. Source `test-fixtures.sh` in your script, then use `use_fake_registry <project_dir> <engine> <fixture_name>` to seed a project with the fixture pinned as `rulesRegistry`.
3. Run the CLI via `assert_cmd_exit <expected> <label> <log_file> -- env -C "$project" node "$CLI" rules <subcommand>`. The `--` separator is literal and required.
4. Assert JSON payload shape via `assert_json_field` / `assert_json_array_length` (jq fast-path when available, `node -e` fallback otherwise). For regression guards that a file was NOT touched, capture the hash before the command with `sha_of` and verify after with `assert_file_unchanged`.
5. If the new scenario introduces a new exit-code assertion, the matrix guard in `test-exit-codes.sh` picks it up automatically via its `assert_exit` / `assert_cmd_exit` grep.
6. Wire new files into `scripts/test-rules.sh` via a new `run_nested_test "<label>" "$SCRIPT_DIR/<file>"` line so `npm test` covers them end-to-end.

### Regression guards baked into the suite

- **Hard guard (`test-rules-sync.sh`)**: `sync --replace --prune` never materializes rules absent from state (mirrors the `if (!existing) continue` guard in `syncRulesState`). Without this guard a routine `unikit-ai update --force` would carpet-bomb a project with every registry rule.
- **No auto-sync (`test-rules-registry.sh`)**: neither `rules registry set <url>` nor `rules registry reset` rewrites rule files on disk. Captured hash before/after each command.
- **Case-insensitive lookup (`test-rules-show.sh` / `test-rules-install.sh`)**: legacy `CODE-STYLE` ids normalize to canonical `code-style` via `normalizeRuleId`.
- **Phase 2 downgrade event (`test-rules-sync.sh`)**: `sync --replace` against a registry with a LOWER version than state emits the `phase2:downgrade` log line so accidental rollbacks never happen silently.
- **createRegistry origin fold (`test-rules-registry.sh`)**: `OFFICIAL_REGISTRY_URL` as a literal collapses `primary`/`official` into one `HybridRegistry` instance so `getResolvedOrigin()` returns `'official'` instead of misstamping every installed rule with `origin: 'primary'`.
- **Exit code matrix (`test-exit-codes.sh`)**: walks the per-command tests and asserts every documented exit code (0, 1, 3, 5, 6, 7 - plus exemptions for 2 and 4) has at least one assertion. Cross-checks the `EXIT` enum in `src/cli/commands/rules.ts` against the contract table above.
